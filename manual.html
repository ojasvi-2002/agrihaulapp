Core concept

How the whole system works

AgriHaul connects three groups of people — your agritech client (ops team), farmers, and truck drivers — using the simplest possible technology: SMS text messages and a web dashboard.

Here is the exact journey of a single dispatch, from farmer's text to truck arriving at the farm:

1
Farmer texts the AgriHaul number
The farmer uses their basic phone (any phone, no smartphone needed) to send a fixed-format message to the dedicated AgriHaul phone number.
READY 500 KG
2
Twilio receives the SMS and fires a webhook
Twilio (the SMS gateway) receives the text instantly and sends an automatic notification (called a webhook) to Make.com, passing along the message text and the farmer's phone number.
3
Make.com parses the message and looks up the farmer
Make.com reads the SMS text, extracts the weight (500), and looks up the farmer in Google Sheets by matching their phone number. It retrieves the farmer's GPS coordinates, name, and village.
4
Haversine formula finds the nearest available truck
Make.com fetches all trucks from Google Sheets where Status = "Available". For each truck, it runs the Haversine formula — a math equation that calculates the real-world distance in km between the farmer's GPS and the truck's GPS. It picks the smallest distance.
5
Dispatch SMS sent to the truck driver
Make.com tells Twilio to send an SMS to the nearest driver's phone number.
DISPATCH: Go to Thiès. Collect 500KG from Amina Diallo. GPS: 14.7833,-16.9240. Reply DONE when loaded.
6
Farmer gets confirmation
At the same time, Twilio sends the farmer a confirmation so they know a truck is coming.
Confirmed Amina. Truck is on the way. Driver: Ibrahim Bah. Reply WAIT to cancel.
7
Truck status updated to "En Route"
Make.com writes "En Route" to that truck's row in Google Sheets so no second dispatch is sent to the same driver.
8
Driver replies DONE — cycle resets
Once the pickup is complete, the driver texts DONE. Make.com sets the truck status back to Available, logs the dispatch in DispatchLog, and the dashboard updates on the next refresh.
DONE
System design

Architecture — the 5 tools

The platform uses exactly five cloud tools. Each one does one job. They connect like a chain — the output of each feeds the next.

📱 Farmer / Driver phones
Any basic phone. Send and receive SMS. No internet or app required.
↓
📡 Twilio / Africa's Talking
SMS gateway. Owns the dedicated phone number. Receives all incoming texts. Sends all outgoing texts. Fires a webhook to Make.com instantly on every SMS.
↓
⚙️ Make.com
The automation brain. 4 visual scenarios: Dispatch, Location Update, Done, On/Off. No code — drag and drop. Reads Google Sheets, runs math, fires Twilio replies.
↓
📊 Google Sheets
The database. 3 tabs: Farmers, Trucks, DispatchLog. Make.com reads and writes. Your ops team edits directly.
🖥️ AgriHaulOps Dashboard
Static website on Netlify. Reads Sheets CSV. Shows map, metrics, logs. Only your ops team uses this.
💡 Key insight

Farmers and drivers never touch the dashboard. The dashboard is only for your client's operations team. Farmers and drivers interact solely via SMS.

Farmers

Farmer SMS guide

Farmers send messages to one number — the AgriHaul dedicated SMS number. They only need to know two commands.

⚠️ Format is strict

Farmers must type the message exactly as shown. The system parses by keyword — if they type "READY500KG" (no spaces) it won't work. Print and hand out the wallet card during onboarding.

🌾 Farmer
READY [weight] KG
Triggers dispatch. Replace [weight] with actual kg. Example: READY 500 KG
🌾 Farmer
WAIT
Cancels a pending pickup before the truck has departed. Send only immediately after the confirmation SMS.

What the farmer receives back

SituationSMS the farmer receives

No truck availableNo trucks available right now. Please try again in 30 minutes.
Format not recognisedFormat not recognised. Please text: READY [weight] KG

Adding a new farmer

Two ways to add farmers to the system:

  1. Google Forms — share the onboarding form with your field staff. Each submission automatically adds a row to the Farmers tab in Google Sheets.
  2. Dashboard — log in to the AgriHaulOps dashboard → Farmers page → click + Add farmer. Fill in name, phone, village, and GPS coordinates.
📍 GPS coordinates

Each farmer needs a GPS latitude and longitude stored in the Farmers sheet. The easiest way: open Google Maps on a phone, press and hold on the farm location, and read the coordinates that appear at the bottom. Format: 14.7833, -16.9240

Drivers

Driver SMS guide

Drivers receive dispatch instructions automatically. They also use SMS to update their GPS location and availability status. Four commands to learn.

🚛 Driver
LOC [lat] [lon]
Updates your GPS position in the system. Send at the start of every shift. Example: LOC 14.6928 -17.4467
🚛 Driver
DONE
Marks pickup complete. Your status resets to Available and you're ready for the next dispatch.
🚛 Driver
OFF
Marks you as unavailable (breakdown, rest, end of shift). No more dispatches until you send ON.
🚛 Driver
ON
Marks you as available again after being offline. System can now dispatch pickups to you.

Driver daily routine

AM
Start of shift — send location
Text your current GPS coordinates so the system knows where you are.
LOC 14.6928 -17.4467
→
Receive dispatch
When a nearby farmer is ready, you'll receive a dispatch SMS with the farmer name, GPS, and weight. Drive to the location.
✓
After loading — send DONE
Once you've collected the produce and it's loaded, text DONE. Your status resets to Available.
DONE
⚠️ Location matters

The nearest-truck algorithm only works if drivers keep their GPS updated. If a driver hasn't sent LOC today, the system uses the last known position which may be from yesterday — causing wrong dispatches. Enforce LOC at shift start.

Auto-refresh

The dashboard automatically re-fetches data from Google Sheets every 30 seconds. You'll see a brief "Data refreshed" toast notification at the bottom right. You can change this interval in config.js → SHEETS.REFRESH_INTERVAL (value is in milliseconds — 30000 = 30 seconds).

Hovering over a pin

Move your mouse over any pin to see a tooltip showing: truck ID, driver name, current status, GPS coordinates, and last updated time. For farmers: name, village, and GPS.

Map bounds

The map is scaled to cover your operating region. Your sheet data covers West Africa (Senegal, Ghana, Côte d'Ivoire, Guinea). If you expand operations, update MAP_BOUNDS in js/map.js.

💡 Pins don't move in real time

Truck positions update when drivers send a LOC SMS. The map then shows the new position on the next 30-second data refresh. This is not live GPS tracking — it's last-known position.

Toggle button

Each truck row has a Toggle button that cycles the status: Available → En Route → Maintenance → Available. Use this to manually override a truck's status from the dashboard — for example, if a driver's phone is dead and they can't text OFF.

⚠️ Toggle is local only

Status changes made via the Toggle button on the dashboard are not written back to Google Sheets — they only affect what you see until the next data refresh. To permanently change a truck's status, edit the Status column directly in the Google Sheet or wait for the driver to send the correct SMS.

Adding a new truck

Click + Add truck (top right). Fill in:

  • Truck ID — use the next sequential ID (e.g. TRK-009)
  • Driver name — full name
  • Driver phone — include country code (e.g. +221 77 ...)
  • Status — initial status (Available or Unavailable)
  • GPS Lat / Lon — starting position (optional, can be updated via LOC SMS)

Also add this truck manually to your Google Sheets Trucks tab so it persists after the next data refresh.

Setup guide

Google Sheets connection

The dashboard reads data from your published Google Sheet CSV URLs. Here's how to set it up:

  1. Open your Google Sheet (use AgriDispatch.xlsx as the template — it has the correct column names)
  2. Go to File → Share → Publish to web
  3. Change "Entire Document" to the Farmers tab
  4. Change the format dropdown from "Web page" to CSV
  5. Click Publish and copy the URL shown
  6. Repeat steps 3–5 for the Trucks tab and DispatchLog tab
  7. Open js/config.js and paste the three URLs into SHEETS.FARMERS_URL, TRUCKS_URL, and DISPATCH_URL
✓ How to test it

After adding the URLs, open the dashboard, log in, and check the browser console (F12 → Console tab). If you see [AgriHaul] No Sheet URL configured it means the URLs weren't saved. If data loads correctly, no console messages appear.

Twilio + Make.com setup

Twilio (SMS gateway)

  1. Sign up at twilio.com (free trial gives $15 credit)
  2. Go to Console → Phone Numbers → Buy a number with SMS enabled (~$1)
  3. Go to Console Dashboard → Account Info. Copy Account SID and Auth Token
  4. Open js/config.js and fill in TWILIO.ACCOUNT_SID, AUTH_TOKEN, and FROM_NUMBER

Make.com (automation)

  1. Sign up at make.com. Create a new scenario.
  2. Add a Webhooks → Custom webhook module. Copy the webhook URL.
  3. In Twilio Console: your phone number → Messaging → A message comes in → paste the Make.com webhook URL
  4. Build the 4 scenarios (Dispatch, LOC update, DONE, ON/OFF) using the prompt chains from the project documentation
  5. Paste the webhook URL into config.js → MAKE.WEBHOOK_URL

GitHub + Netlify deployment

The dashboard is a folder of files. Push to GitHub once — Netlify deploys automatically every time you push a change.

First time: push to GitHub

# 1. Open terminal and go to the project folder cd agrihaulapp # 2. Initialise git and make first commit git init git add . git commit -m "AgriHaul ops dashboard" # 3. Create repo on github.com, then connect git remote add origin https://github.com/YOUR_USERNAME/agrihaulapp.git git branch -M main git push -u origin main
⚠️ config.js stays off GitHub

js/config.js is in .gitignore — it will NOT be committed. This is intentional. Your Twilio keys and Sheet URLs must never be in a public GitHub repo. Only config.example.js (with empty values) is committed as a reference.

Connect Netlify

  1. Go to netlify.com → sign in with GitHub
  2. Click Add new site → Import an existing project → GitHub
  3. Select your agrihaulapp repo
  4. Leave build command and publish directory blank (it's plain HTML, no build step needed)
  5. Click Deploy site — live URL appears in ~30 seconds

Updating the dashboard later

# Make your changes, then: git add . git commit -m "Updated map bounds / fixed farmer table" git push # Netlify auto-deploys within 30 seconds
💡 Custom domain (optional)

In Netlify: Site settings → Domain management → Options → Edit site name. Type something like agrihaulops to get https://agrihaulops.netlify.app for free. Or point a custom domain you own (e.g. agrihaulops.com) by adding a DNS record.

Help

Troubleshooting

ProblemMost likely causeFix
Farmer sends READY but driver gets nothing All trucks are set to En Route or Maintenance, OR Make.com scenario is paused Check Trucks sheet — at least one must show "Available". Check Make.com scenario is active (green toggle).
Wrong truck dispatched (far away) Driver hasn't updated GPS today. System uses stale coordinates. Driver must send LOC [lat] [lon] at start of every shift.
Dashboard shows old data Google Sheets URLs not configured, or Sheet not published to web Check config.js URLs. Open a published URL directly in browser — it should return CSV text.
Dashboard map shows no pins Lat/Lon values in sheet are empty or text instead of numbers Check the Trucks and Farmers sheets — Lat and Lon must be numbers, not text.
Login fails Credentials don't match config.js values Open js/config.js and verify DEMO_USERNAME and DEMO_PASSWORD.
Netlify shows old version after push Netlify deploy still in progress Wait 60 seconds, then hard refresh the browser (Ctrl+Shift+R). Check Netlify dashboard for deploy status.
Make.com scenario stopped working Free Twilio trial credit ran out, or Make.com operation limit hit Check Twilio billing console. Check Make.com scenario history for error messages.
📋 Before contacting support

Open the browser console on the dashboard (press F12 → Console tab). Any errors show there with a description. Also check Make.com → your scenario → History — each run shows success or failure with the exact error message.
